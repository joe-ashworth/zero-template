Development Cycle & Workflow

  • Identify project stage to determine workflow. Initial Design, Build-out, Production.
  • Currently refining a process that replaces Dreamweaver with Brackets. WAMP works great for me where XAMPP has been giving me problems when modifying http.conf to change project location. Brackets has a buit-in previewer that is amazing, but falls short of Prepros once pages are sliced into php includes.
  • WAMP modification: Configure the http.conf file within Apache in WAMP to point to root of all project files. For me, all of my sites are in folders within a folder named sites. So I changed WAMP document root in http.conf (in 2 places) from the default install location to F:/sites. Then I can view a project by going to http://localhost/sitefoldername.

Project Stages

  • Initial Design: This is the barebones creation of a home page and a single subpage to be duplicated. It is done entirely in Brackets using its preview tool. Once a good home page and sub page have been completed, it's time to begin Build-Out
  • Build-Out: This stage is for extracting html into php includes and duplicating pages as needed. Now that pages are php, we move to WAMP and Prepros and discontinue using Brackets' preview. Prepros is configured to run its preview at the localhost address from WAMP.
  • Pre-Production: When all pages are in place with dummy content, open the site in Dreamweaver to enter actual content. Dreamweaver is superior for entering and tweaking text. Its preview pane works great by wrapping text in p tags or entering link hrefs from the properties panel.
  • Production: When the site is live, it is an official production site. Dreamweaver is used to FTP upload directly and make changes as needed.

General suggestions for markup and styling that make life easier

  • Avoid static widths whenever possible. Use percentages.
  • Try to use padding to move things around whenever possible.
  • Try to avoid adding left or right margins or padding to block elements (mostly containers). They add size to the element. A div that is 25% wide with a 1% margin (or padding) actually uses 27%. Pad the contents instead if necessary.
  • Try to avoid margins on containers unless you accomodate for the behavior of margins from 2 different elements touching. Learn the displacement differences between padding and margins to avoid strange problems
  • Floats can be cleared automatically using the .group style on the parent container (see stylesheet).
  • Try to get acquainted with using em instead of px on text or most anywhere possible. pixels are rigid, ems fluctuate with text size. Style the body element with a text size or simply... text-size:100%;
  • Use a set of classes for columns that define width. Using .one-fourth, .one-third, .two-thirds, .one-half, etc. throughout the site gives you a way to tweak breakpoints across the whole site quickly.

What I'm Working On

I'm also starting to focus on true semantic markup. This means using article, header, footer, nav, section, and aside tags more freely. Since I don't attach any styles to divs specifically, it doesn't matter what I use to replace them (style-wise).

I'm building a replacement for grids. Frameworks and Boilerplates are full of unnecessary code and nested divs that are just ugly, but I need a meaningful way to make multiple columns size properly under changing widths, so I'm writing a healthy block of css that will manage 3 variants of floating columns with quarters, thirds, halves and maybe fifths and sixths.
